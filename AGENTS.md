# Phomo

Android SIP client (Kotlin, single `:app` module). Minimalist outbound calling
over SIP — built for calling overseas cheaply through a Twilio SIP trunk while
handing domestic calls back to the native dialer / SIM. Targets Android 16+ on
Pixel and Samsung at a minimum. Uses Android platform APIs and conventions
(Telecom framework, Material 3) and stays out of the way.

## Project documentation

- Keep `SPEC.md` up to date when changing product behavior, architecture, the SIP/registration model, the Telecom integration, persistence, permissions, navigation, or testing strategy.
- **`SPEC.md` records product, functionality, and architecture decisions — not low-level implementation detail.** It captures *what* Phomo does and *why* a design was chosen (when a dialed number is routed to SIP vs. handed back to the SIM, the on-demand registration lifecycle and why there's no persistent service, how the self-managed `ConnectionService` and `CallRedirectionService` divide responsibility), so a reader can understand and QA the product from the spec. It is not the place for facts that live in the code and change with it: exact dp values, private helper names, per-locale string wording, which composable holds a flag, the exact SIP header a request carries, or whether a given string is translated. Ask "would this still be true and worth stating if the implementation were rewritten?" — if not, leave it in the code and its comments. When a change only touches that kind of detail, don't add a `SPEC.md` line for it; when a change does move the product or architecture, state the decision at that altitude and stop.

## Engineering quality bar

- Every change must hold the line on **correctness, battery efficiency, and call reliability & audio quality**. This is a phone dialer's substitute — when the user taps Call, a real call to a real person has to connect, sound clean both ways, and end cleanly, and the app must not drain the battery while idle in the user's pocket. A regression in any of the three is immediately and personally visible: a dropped call, an echo, a one-way-audio bug, or a phone that's flat by lunchtime destroys trust in a way a cosmetic glitch never would. Treat these three as hard acceptance criteria, not aspirations.
  - **Correctness**: the change matches `SPEC.md` and the user's stated intent, handles the obvious edge cases (no network / cellular-only / captive Wi-Fi, permission denied, SIP registration failure or timeout, the far end rejecting or not answering, an incoming GSM call arriving mid-SIP-call, airplane mode, a domestic number that must fall back to the SIM, a malformed or unreachable SIP registrar), and preserves existing invariants. Routing decisions especially: a number that should go to the SIM must **never** be silently swallowed by Phomo, and a divert that fails must fall back to the native path rather than dropping the call. New behavior is covered by a unit test; when fixing a bug, add a test that fails before the fix and passes after — otherwise the regression will return.
  - **Battery efficiency**: this is a first-class product constraint, not a nice-to-have. Phomo is outbound-only in v1 specifically so it can avoid a persistent foreground SIP registration and the wakelocks/keep-alives an always-registered client needs. The default posture is **register on demand at call time, tear down completely when the call ends** — no background service, no periodic re-REGISTER, no push socket held open while idle. Any change that adds background work (a service that outlives a call, a `WorkManager` job, a wakelock, a network keep-alive, a `ConnectivityManager` callback that stays registered) must justify why it can't be done on-demand, must be scoped to the call's lifetime, and must be called out explicitly in the PR. When in doubt, do nothing in the background.
  - **Call reliability & audio quality**: signaling and media must be robust on real mobile networks. NAT traversal (ICE/STUN/TURN as the provider requires), codec negotiation, DTMF, echo cancellation, and clean teardown are all part of "done" — a call that connects but has one-way audio, echo, or a 3-second delay before ringback is not done. The audio focus / routing / Telecom handshake must be correct so the call interacts properly with the earpiece/speaker/Bluetooth, the proximity sensor, and any concurrent cellular call. Prefer the provider's and the SIP stack's proven defaults over hand-rolled media handling.
- When you cannot verify one of these three locally (the sandbox has no cellular radio, no microphone, no real SIP peer, and can't measure battery), say so explicitly in the chat update — e.g. "verified routing logic by unit test, verified the registration teardown path by inspection, could not place a real call or measure battery because the sandbox has no radio/mic/SIP peer" — rather than implying all three were checked. The user needs to know which pillar is still owed a real-device call test. **A change to signaling, media, audio routing, or the registration lifecycle is not truly verified until a real call has been placed on a real device**; flag those for on-device testing every time.

## Spacing, padding, and margins

- **Measure padding and margins on every UI change and ensure they are even, consistent, and predictable.** Phomo's UI is small but it's a Compose surface the user looks at every time they place a call, and Compose makes it easy to introduce spacing drift one composable at a time. Treat spacing as part of the deliverable, not a polish pass.
  - **Stick to a 4dp grid.** Every padding / margin / spacing value should be a multiple of 4dp (`4`, `8`, `12`, `16`, `20`, `24`, `32`, `40`, `48` ...). The most common values are `4`, `8`, `12`, `16`, and `24`. If you reach for `5`, `7`, `10`, `14`, etc., stop and pick the nearest grid value, or surface the off-grid value to the user with a one-line explanation of why it's necessary (baseline alignment, hairline divider). There is no spacing-token file yet (`PhomoTheme.kt` does not export `Spacing.*`), so the discipline is enforced by inspection — `grep -nE "\\.dp\\b" app/src/main/java/dev/phomo/ui/<file>.kt` on any file you touch and confirm every value is on the grid.
  - **Reuse the values already in use.** Before introducing a new dp value to a sibling element, check the surrounding composable and its peers for the values they already use, and match them. Gratuitous variation between `14.dp`, `16.dp`, `18.dp` on visually similar elements is the exact inconsistency this rule targets.
  - **Symmetry by default; asymmetry only with a reason.** `Modifier.padding(horizontal = 16.dp, vertical = 8.dp)` is the right default for a labelled row. Every asymmetry needs a one-sentence justification in the PR / chat. No silent asymmetry.
  - **Verify visually, not just by reading dp values.** After any spacing change, look at the Roborazzi screenshot diff for the affected screen and confirm the layout is visually even. When the sandbox can't run screenshot tests, say so and ask the user to eyeball the device build.
- **Flag flakes and inconsistencies you spot, even outside the diff.** While editing a file you'll notice a neighbouring composable using off-grid or inconsistent spacing. Surface every such case to the user with a one-line note — file, line, what's off, and a proposed fix — even when it's outside the current task. Don't silently fix them in the same commit (that turns a one-line PR into a sprawling refactor), and don't ignore them. The user decides whether to file a follow-up, fold the fix in, or leave it.

## Git workflow

- Always start work from the latest `origin/main`. Before the first commit on any feature branch, run `git fetch origin main` and rebase the working branch onto `origin/main` (`git rebase origin/main`). This applies even when the branch already exists — a stale branch must be brought current before new work is added.
- If the rebase produces conflicts, resolve them rather than abandoning the rebase or branching from an older base.
- Never push commits authored on top of an out-of-date base when a fast-forward rebase onto `origin/main` was possible.
- Clean up the unmerged commit history before requesting review and again before merge. Iteration leaves `fix CI`, `address review`, `wip`, `fix typo`, and reverted-experiment commits scattered through the branch; before asking for review (and again right before merge) squash/rebase via `git rebase -i origin/main` so every commit on the PR is a meaningful step toward the final change. The repo rebase-merges, so every commit on the PR is replayed onto `main` individually — each keeps its own subject (which ships to "What's new", see "Commit messages"), its own `git blame` lines, and its own bisect step. A messy branch ships a messy `main`; the cleanup is on you. After rewriting, force-push with `git push --force-with-lease` (never bare `--force`). When individual commits on the branch have already been individually reviewed, ask before rewriting so reviewers don't lose the comment-to-commit anchors.

## Commit messages

- **Every commit subject in a push to `main` ships to the Play Store internal track as part of that release's "What's new" entry.** `.github/workflows/android-ci.yml` walks every commit between the previous and current `main` tip, skips commits whose subject starts with a non-user-facing prefix (`ci:`, `docs:`, `internal:`, `refactor:`, `test:`/`tests:`) and pure docs / dotfile housekeeping commits, and emits the surviving subjects oldest-first into `whatsnew-en-US` as a `• `-bulleted list — always bulleted, even when only one release-worthy commit is in the push. Commit bodies are always dropped. So write every subject for end users, not for the next engineer reading `git log`.
  - **Use a non-user-facing subject prefix to keep a commit out of "What's new".** A commit with no user-visible change should start with one of these prefixes; CI then drops it from the release notes (it still runs the full build-and-test job — the prefix only suppresses the release-notes entry):
    - `ci:` — CI / workflow plumbing (e.g. snapshot regenerations, action bumps).
    - `docs:` — documentation only (`AGENTS.md`, `SPEC.md`, `README`, `docs/*`). `PRIVACY.md` is the exception — it backs the hosted privacy policy linked from the About dialog, so it counts as user-facing.
    - `internal:` — build configuration, dependency upgrades, and other plumbing with no user-visible effect.
    - `refactor:` — behavior-preserving code changes (renames, extractions, reorganizations).
    - `test:` (or `tests:`) — test-only changes.

    Use these precisely: the prefix is a promise that the commit has **no user-visible effect**. If the commit also changes production behavior, it is a feature or fix commit — title it for end users.
  - **Sentence case, plain English, no jargon.** "Fall back to the SIM when a SIP call can't connect" — not "Wire ConnectionService onCreateOutgoingConnectionFailed to Telecom fallback." Avoid internal symbol names (`ConnectionService`, `RoleManager`, `REGISTER`), avoid mechanism-level detail, avoid commit-style imperatives that read as engineering notes ("Gate", "Arm", "Wire").
  - **Keep it ≤ ~70 characters** so it isn't truncated in the Play Console preview or in `git log --oneline`. CI measures the full formatted release notes against Play's 500-char per-language cap and drops trailing subjects when a push overflows.
  - **Engineering detail goes in the body.** The full mechanism, the bug it fixes, the file:line of the regression — all of that belongs after the blank line.
  - **Rebase-merging a PR**: each commit's own subject lands on `main` and ships to "What's new" — the PR title is *not* used as a commit subject. Title **every** commit on the branch for end users, not just the PR title.

## Working with PRs

- Use the `mcp__github__*` MCP tools for all GitHub operations. The `gh` CLI is not available in this sandbox.
- Never leave a review comment thread silently dismissed. Either reply on the thread or resolve it. When you think a comment is a false positive, say why on the thread (one or two sentences): the reasoning is what the user wants surfaced. Acknowledgement noise ("good catch, will do") is fine and preferred over silence; the discipline is "say something or resolve," not "say nothing."
- Always link every open PR in the stack. Any time you push, summarise CI, or invite the user to review, list every currently-open PR on the feature by URL — one per line — not just the topmost one. The Claude Code mobile UI only renders the first PR card in a message and treats later links as plain text.
- Refresh the PR title and body on every push. After every push, update the PR via `mcp__github__update_pull_request` so the title and body describe the **full and latest state of the branch** — not the initial scope. A stale title misleads reviewers and the PR view (the commit subjects, not the PR title, ship to "What's new"), and a stale body misdirects reviewers and future bisecters. After `git push`, re-read `git diff origin/main...HEAD` and patch whichever no longer matches.
- Always include visual diff links in PRs for UI/screenshot-affecting changes. If a Roborazzi/snapshot image changes, generate a before/after/diff artifact from `origin/main` versus the branch using the same screenshot path, prefer a single PNG contact sheet (`before | after | changed pixels`), place it under `/opt/cursor/artifacts`, and embed it in the PR body with an `<img>` tag. Restore any generated snapshot files before committing unless the snapshot update itself is intentional.
- Report when Copilot finishes reviewing a fresh push. Once its review event lands for the latest commit, surface a one-liner naming the SHA and comment count — e.g. `Copilot reviewed 87d9f02 — 0 comments`. Tie it to the latest pushed SHA so a stale review of a superseded commit isn't conflated with the current state.
- Report Android `versionCode` after every merge to main. When a PR merges, fetch main and run `git rev-list --count origin/main` to get the `versionCode` (`app/build.gradle.kts` derives it from this count). Report it as e.g. `Need versionCode 21 (b81c23d) or higher to test PR #12's fallback fix` — number, short SHA, and a one-clause summary of what the change gates. Sandbox clones are usually shallow (`git rev-parse --is-shallow-repository` returns `true`), which silently truncates `rev-list --count`. Run `git fetch --unshallow origin main` once at the start of any session that will report `versionCode`s.
- Keep watching merged PRs for late review comments. Reviewers and bots routinely comment after merge. Stay subscribed to the PR's activity after the merge and handle each new comment per the "say something or resolve" rule. Stop watching once every comment posted on or after the merge commit has been answered or resolved, or after ~24h of silence, whichever comes first.
- Skip echo events silently. The `mcp__github__add_reply_to_pull_request_comment` / `add_issue_comment` / `add_comment_to_pending_review` tools post under whichever GitHub identity backs the MCP auth, so a moment after I post a reply the same body comes back as a `<github-webhook-activity>` event authored by that same identity. That is the echo of my own reply, not user feedback — skip it silently. The test is "did *I* just post this body?" not "who is the author?" Real review comments from the same identity — ones I didn't author this turn — still need the usual reply-or-resolve handling.
- **On any CI failure, diagnose in this order:**
  1. **Check for a PR comment first.** The `build` job posts failing JUnit XML snippets as a PR comment when `./gradlew test` exits non-zero. If a comment appeared, read it — it has the exact test class, method, and stack trace.
  2. **No PR comment ≠ compile error.** Several `build` failures produce no JUnit XML and therefore no PR comment: (a) `assembleDebug` Kotlin compile error; (b) `gradle test` Kotlin compile error in main *or* test sources; (c) `gradle lint` failure (e.g. `MissingTranslation`); plus the AAPT2 / `mergeDebugResources` step inside `assembleDebug` that rejects malformed resource XML. Treat "no PR comment" as "the failure is somewhere before tests ran" — widen the suspect list to lint and resource-merge before assuming Kotlin compile.
  3. **`screenshot-tests` never posts PR comments.** That job prints failures to the Actions log only. When it is the *only* failing job and there is no PR comment, check whether `build` is green — if so the failure is in the screenshot suite, not a compile error.
  4. **Reproduce the merge locally before bisecting your own branch.** The pull-request `build` job checks out `refs/pull/<N>/merge` (an auto-generated merge of head + base), so it builds your branch merged with the current `origin/main`. A clean `git merge origin/main --no-commit --no-ff` followed by `grep -rn "<old-symbol>" app/src/` will surface API renames or removed callsites that look fine on your branch alone but break in the merged build.
  5. **Use the artifact list as a coarse stage signal.** The screenshot job uploads `roborazzi-screenshots` unconditionally but `screenshot-test-reports` only when JUnit XMLs exist. The `build` job uploads `unit-test-reports` only when JUnit XMLs exist. So: only `roborazzi-screenshots` and no `screenshot-test-reports` ⇒ the screenshot job died before any test ran (compile error); a `screenshot-test-reports` artifact exists ⇒ tests ran and at least one failed.
  6. **Fetch artifacts as a last resort.** Pull the run ID from the webhook's `details_url`, hit `https://api.github.com/repos/<owner>/<repo>/actions/runs/<runId>/artifacts`, follow `archive_download_url`, and read `TEST-*.xml`. A 403 is the anonymous rate limit — fall back to code inspection.
  7. **Before debugging, check whether the failure is pre-existing.** Run `mcp__github__pull_request_read` with `get_check_runs` for the PR's *base* commit to see if the job was already red. A pre-existing failure is not yours to fix; note it and move on.

## Language and spelling

- **Use US English everywhere — most importantly in the base English strings and in commit subjects.** `app/src/main/res/values/strings.xml` is the source-of-truth English locale, and it maps to American English (`en-US`) on the Play Store; commit subjects flow straight into the `whatsnew-en-US` "What's new" entry. Prefer the American form: `color` not `colour`, `customize`/`organize`/`recognize` not `-ise`, `behavior` not `behaviour`, `center` not `centre`, `canceled`/`canceling` not `cancelled`/`cancelling`, `gray` not `grey`, `dialog` not `dialogue`, `license` (noun and verb) not `licence`. This covers user-facing string values **and** commit subjects/bodies, PR titles/descriptions, code comments, KDoc, and identifiers. It does **not** override platform or third-party API spellings: Android's own `android:color`, `Color`, etc. stay as the framework spells them.
- This rule is about US-vs-UK *spelling*, not about adding new locales. Don't create a `values-en-rGB/`. Hold `SPEC.md`, `README`, and any other prose to the same American-spelling bar.

## Translations

- Confirm the English copy with the user before translating it. Don't write translations in any `values-*/strings.xml` until the corresponding English string in `app/src/main/res/values/strings.xml` — or the proposed wording change — has been explicitly approved.
- **Ship English first, translations in a second PR — never the same PR.** The working order is:
  1. **English-only PR.** Add the new entry to the base `app/src/main/res/values/strings.xml` only, and *do not* touch any `values-*/` locale. Suppress the `MissingTranslation` lint warning *per string* with `tools:ignore="MissingTranslation"` on the `<string>` element (the root `<resources>` needs `xmlns:tools="http://schemas.android.com/tools"`), and leave a `<!-- TODO: translate -->` comment on the line above so the untranslated key is greppable.
  2. **Translation PR.** Once the English wording is approved and merged, open a separate follow-up PR that adds the approved string to every existing `values-*/` locale and removes the `tools:ignore` and the `TODO` comment.
- Sequence the English-copy approval *in chat*, not via a commit: propose the wording in conversation and wait for explicit user approval before opening the English-only PR.

## Remote build environments (Cursor Cloud and Claude Code on the web)

This project is worked on from more than one managed remote environment. The toolchain facts below apply to all of them unless a bullet calls out a difference.

### Environment

- **JDK 21** is pre-installed on the VM. AGP 9.2.0 requires JDK 17+.
- **Android SDK location** is `/opt/android-sdk` (`ANDROID_HOME` / `ANDROID_SDK_ROOT`). How it gets there differs by environment:
  - **Cursor Cloud**: the SDK is pre-installed, with `ANDROID_HOME`, `JAVA_HOME`, and `PATH` set via `~/.bashrc`. In non-login shells you may need to `source ~/.bashrc` or export these explicitly before running Gradle.
  - **Claude Code on the web**: the container starts with `ANDROID_HOME` set but the SDK *not yet installed*. The `SessionStart` hook at `.claude/hooks/session-start.sh` provisions it (command-line tools + `platforms;android-36` + `platform-tools`, licenses accepted) at the start of every web session. If you ever find `/opt/android-sdk` empty mid-session, run that hook manually (`CLAUDE_CODE_REMOTE=true .claude/hooks/session-start.sh`).
- The Gradle wrapper (`./gradlew`) auto-downloads Gradle 9.4.1 on first run.
- AGP auto-installs `Android SDK Platform 36.1` (compileSdk minor API level 1) on the first build if only `platforms;android-36` is present — which is what the hook seeds.

### Testing expectations

- Code changes must include or update relevant unit tests. Routing decisions (SIP vs. SIM), number classification (domestic vs. international / E.164 handling), and the registration state machine are pure logic and must be unit-tested exhaustively — these are exactly the parts that fail silently and lose a call.
- UI changes must include or update screenshot tests that cover the changed UI state — and the test class must be wired into `.github/workflows/android-ci.yml`. The screenshot job runs `:app:testDebugUnitTest` under `-Proborazzi.test.record=true` against an explicit `--tests` allow-list (one `run:` step per screenshot test class); a new `*ScreenshotTest` class that isn't on that list never records snapshots in CI even when it passes locally. After adding a new screenshot test, also add a `Run … screenshot tests` step that targets it. The snapshot path is `app/src/test/snapshots/images/`. The screenshot job auto-commits recording drift back to same-repo PR branches (`ci: refresh recorded screenshots`), so after CI runs on a UI-touching push, `git pull` before pushing again or a `--force-with-lease` will be refused.
- Whenever a commit touches UI, post a visual diff back to the user. After CI completes, pull the branch (the screenshot job may have pushed a `ci: refresh recorded screenshots` commit) or the `roborazzi-screenshots` artifact, locate every `app/src/test/snapshots/images/*.png` whose blob hash differs from `origin/main`, build a `before | after | diff` contact sheet, save under `/opt/cursor/artifacts`, and reply with embedded `<img>` links. For brand-new screenshots, post an "added" image.
- **Real-call verification is owed for anything touching signaling, media, audio routing, registration, or routing.** The sandbox has no radio, no microphone, and no SIP peer, so those paths can only be verified on a real device with a real (Twilio) SIP trunk. Say so explicitly and flag the change for on-device testing — do not imply a call path was verified when it was only inspected.
- Run the relevant tests before submitting changes. Network access depends on the environment: Claude Code on the web (with the `SessionStart` hook) can reach Google Maven and run the full build/test/lint locally; in Cursor Cloud, Google Maven is sometimes blocked. When you *can't* run Gradle, go through the pitfalls below by inspection; when you *can*, actually run `./gradlew test` and `./gradlew lint`.

### Common pre-CI pitfalls

Each item below produces an opaque CI failure (no PR comment, fast-failing job) — diagnose by inspection rather than by re-running CI.

- **Don't invent Compose Test imports.** Several `SemanticsNodeInteraction` API points (`assertExists`, `assertDoesNotExist`, `assertIsDisplayed`) are member functions, *not* extensions, so writing `import androidx.compose.ui.test.assertDoesNotExist` produces an "Unresolved reference" compile error even though the call site works without an import. Rule: when adding a Compose test API call, only add the import if a sibling test already imports it (`grep "import androidx.compose.ui.test.<name>" app/src/test`); otherwise call it as a method.
- **Escape `'` in non-English string resources.** Android resource XML treats a bare apostrophe inside a `<string>` value as a syntax error — `mergeDebugResources` fails with `Apostrophe not preceded by \\` before tests or lint run, surfacing only as an opaque `assembleDebug` failure. French (`l'application`, `d'origine`), Italian (`l'app`), Catalan, and any other locale with a glottal stop need `\'`. After adding any translation, grep the new entries for an unescaped `'`.
- **When adding a required parameter to a `@Composable`, patch every callsite including previews.** Run `grep -rn "<ComposableName>(" app/src/` (the broader `app/src/`, not just `app/src/test/`) to enumerate every callsite — production, `@Preview` functions, and tests — then add the new argument to each. A missed preview callsite compiles fine in tests but fails `assembleDebug` with `None of the following candidates is applicable`.
- **A `TextField` inside a Compose `Dialog` popup window doesn't settle under Robolectric.** Hosting a `TextField`/`OutlinedTextField` inside `androidx.compose.ui.window.Dialog` (which `material3.AlertDialog` uses internally) makes `composeRule.setContent { … }` blow past Compose's idle timeout during initial composition. Factor the body into `@Composable internal fun XContent(…)` and have `XDialog(…)` wrap it in `Dialog { Surface { XContent(…) } }`; production keeps the popup wrapper, and the Robolectric screenshot test composes only `XContent(…)`. Cover the open-from-dialog branch via the ViewModel layer instead of an integration-style test.

### Emulator and connected tests

- **Android Emulator** and a **`system-images;android-36;google_apis;x86_64`** image are available, but **KVM is not** in either remote environment (`/dev/kvm` does not exist). Without KVM the emulator falls back to software emulation (TCG/QEMU), which is extremely slow and may not boot reliably, so `connectedDebugAndroidTest` / `installDebug` are **not practical** in the cloud VM.
- To create an AVD for local or KVM-enabled use:
  ```sh
  avdmanager create avd -n "pixel_api36" -k "system-images;android-36;google_apis;x86_64" -d "pixel_6"
  emulator -avd pixel_api36 -no-window -no-audio -gpu swiftshader_indirect
  ```
- The emulator has no cellular modem and no real audio peer, so it cannot place a real SIP call end-to-end either. Real-call verification means a physical device with a real SIP trunk.

### Notes

- Phomo is a client app with no backend of its own, but it is **not** self-contained the way a launcher is: it depends on an external SIP service (a Twilio SIP trunk / termination) and on the device's microphone, network, and Telecom framework. Those dependencies can't be exercised in the sandbox — see the real-call verification note above.
- The lint report is written to `app/build/reports/lint-results-debug.html`.
- First build takes ~1-2 minutes due to Gradle daemon startup and dependency downloads; subsequent builds are much faster.
